# 하네스 엔지니어링
> AI 에이전트가 실수했을 때, 프롬프트를 고치지 마세요. **하네스(harness)를 고치세요.**
그 실패가 구조적으로 반복 불가능하도록 시스템을 바꾸는 것 — 그게 하네스 엔지니어링입니다.
> 

---

## 1. 왜 하네스가 필요한가 — Why 루프

### AI 에이전트는 실수한다

LLM은 본질적으로 확률적이다. 같은 프롬프트를 줘도 10번 중 9번은 맞고 1번은 틀린다. 프롬프트를 아무리 정교하게 다듬어도 그 1번을 0으로 만들 수 없다. "하지 마"라고 부탁하는 거지, 물리적으로 못 하게 막는 게 아니기 때문이다.

### 실수의 폭발 반경이 달라졌다

챗봇 시대에는 괜찮았다. 답변이 이상하면 사람이 무시하면 됐다. 하지만 지금 에이전트는 코드를 쓰고, 커밋하고, 배포까지 한다. 1번의 실수가 프로덕션 DB를 날리거나, 보안 취약점을 심거나, 의존성을 엉망으로 만들 수 있다. **자율성이 올라간 만큼 실수의 폭발 반경도 커졌다.**

### 사람이 매번 검토하면 에이전트의 의미가 없다

3명이 5개월간 AI만으로 제품을 만든 OpenAI 사례처럼, 에이전트의 가치는 사람이 손 안 대도 돌아가는 데 있다. 매번 사람이 확인해야 한다면 그건 코파일럿이지 에이전트가 아니다.

### 프롬프트는 "부탁"이고, 하네스는 "구조"다

린터가 걸려 있으면 규칙 위반 코드는 물리적으로 머지가 안 된다. 도구 경계가 설정되어 있으면 `DROP TABLE`은 아예 실행이 안 된다. 부탁이 아니라 차단이다. 실수가 발생한 후 고치는 게 아니라, **실수가 발생할 수 없는 환경을 만드는 것**이다.

> **핵심:** AI의 능력은 충분히 강해졌는데 신뢰성은 아직 부족하고, 그 간극을 메우는 게 하네스다.
> 

---

## 2. AI 활용 방법론의 3가지 축

AI를 활용하는 방법론은 3가지 축으로 나뉜다. 순서대로 졸업하는 게 아니라 **전부 다 필요한 상호보완적인 축**이다.

| 축 | 핵심 |
| --- | --- |
| **Prompt Engineering** | AI에게 말을 잘 거는 기술 |
| **Context Engineering** | AI에게 필요한 정보를 적절히 제공하는 기술 |
| **Harness Engineering** ⭐ | AI가 실수할 수 없는 환경을 만드는 기술 |

---

## 3. 하네스 엔지니어링의 구조

원본 영상에서는 4가지 기둥으로 분류했지만, 역할로 다시 묶으면 더 명확해진다. 특히 원본의 "피드백 루프" 기둥은 성격이 다른 두 가지 — 규칙 진화와 쓰레기 수거 — 를 하나로 묶고 있었다.

### 역할로 재편한 하네스 구조

| 역할 | 하는 일 | 방향 |
| --- | --- | --- |
| **컨텍스트** | AI가 뭘 보고 뭘 따를지 정의 | 입력 제어 |
| **가드레일** | 규칙 위반을 실행 시점에 차단 | 지금 막는 것 |
| **진화** | 실수에서 배워서 가드레일 강화 | 다음에 막을 것을 만드는 것 |
| **위생** | 에이전트가 남긴 기술 부채 자동 정리 | 이미 벌어진 엉망을 치우는 것 |

진화와 위생은 서로 방향이 반대다. 진화는 **"앞으로 더 잘하게 만드는 것"**이고, 위생은 **"이미 쌓인 쓰레기를 치우는 것"**이다.

---

### 컨텍스트 — 기계가 읽는 설정 파일

`CLAUDE.md`, `AGENTS.md`, `.cursorrules` 같은 파일들은 단순한 문서가 아니라 **AI가 실행하는 런타임 설정 파일**이다.

- `CLAUDE.md`에 "새로운 라이브러리를 도입하지 마. DB 쿼리는 반드시 ORM을 통해서만 해."라고 쓰면, AI 에이전트는 이 규칙을 자신의 행동 제약으로 인식한다.
- 매번 프롬프트에 반복할 필요가 없다.
- 프로젝트 루트, 서브디렉토리, 개인 설정 등 계층적으로 관리할 수 있다.

**잘 쓰인 CLAUDE.md는 하네스의 출발점이다.** 여기에 프로젝트의 아키텍처 결정, 금지 사항, 코딩 컨벤션, 테스트 정책 등을 명시하면 에이전트가 매 작업마다 자동으로 참조한다.

---

### 가드레일 — 실행 시점의 자동 차단

CI/CD 게이트와 도구 경계를 합친 것이다. 둘 다 **"지금 이 순간 규칙을 어기면 물리적으로 막는다"**는 같은 역할을 한다.

**CI/CD 게이트:**

- **린터(Linter)** — 코드가 규칙을 어기면 자동으로 에러를 띄움
- **구조 테스트** — 의존성 규칙을 테스트로 강제 (예: "이 모듈은 저 모듈을 import하면 안 된다")
- **Pre-commit Hook** — 코드를 커밋하기 전에 자동으로 검사
- **타입 체커** — TypeScript strict 모드, mypy 등으로 타입 안전성 보장

> CI가 실패하면 에이전트가 스스로 수정한다. 사람이 개입하지 않아도 된다.
> 

**도구 경계:**

AI 에이전트가 **어떤 도구를 쓸 수 있고, 어디까지 접근할 수 있는지**를 명확하게 제한한다.

| 영역 | 허용 | 차단 |
| --- | --- | --- |
| 파일 시스템 | `src/` 읽기·쓰기 | `config/` 쓰기 불가 |
| API | 내부 API 호출 | 외부 서비스 호출 불가 |
| 데이터베이스 | `SELECT` 가능 | `DROP TABLE` 절대 불가 |
| Git | 브랜치 생성·커밋 | `main` 직접 푸시 불가 |

**프롬프트는 부탁이고, 가드레일은 물리적 차단이다.** 이것은 최소 권한 원칙(Principle of Least Privilege)의 에이전트 버전이다.

---

### 진화 — 실수가 새로운 규칙이 된다

에이전트가 실수할 때마다, 그 실수는 새로운 가드레일이 된다. 린터 규칙이 추가되고, 테스트가 추가되고, 제약이 추가된다.

- 새로운 종류의 실수 발견 → 린터 규칙 추가
- 반복되는 아키텍처 위반 → 구조 테스트 추가
- 예상 못한 도구 남용 → 도구 경계 강화
- 컨텍스트 파일 미참조로 인한 실수 → `CLAUDE.md` 규칙 보강

> 실수 → 규칙 추가 → 같은 실수 구조적 차단 → 새로운 실수 발견 → 규칙 추가...
> 

이 사이클이 반복되면서 하네스는 점점 더 촘촘해진다. 하네스가 정적인 설정이 아니라 **진화하는 시스템**인 이유다.

---

### 위생 (가비지 컬렉터) — 에이전트가 남긴 쓰레기를 치운다

에이전트가 코드를 빠르게 대량 생성하면 필연적으로 기술 부채가 쌓인다. 사람이 짜도 생기는 문제인데, 에이전트는 속도가 빠르니까 쓰레기도 빠르게 쌓인다.

- 사용하지 않는 함수·변수 자동 제거 (dead code elimination)
- 중복 코드 발견 및 리팩토링 PR 자동 생성
- 불필요한 import 정리
- 테스트 커버리지 하락 감지 및 알림

진화가 **"다음에 더 잘하게 만드는 것"**이라면, 위생은 **"이미 벌어진 엉망을 주기적으로 청소하는 것"**이다. 둘 다 없으면 코드베이스는 시간이 갈수록 부패한다.

---

## 4. 코드로 보는 하네스 (Pseudo-Code)

AI를 단순한 챗봇이 아니라 **"능력은 뛰어나지만 신뢰할 수 없는 함수"**로 생각하고, 엄격한 가드레일로 감싸야 한다.

하네스의 네 가지 핵심 역할:

1. **AI가 볼 수 있는 것을 제한** → 컨텍스트
2. **AI가 해야 할 일을 통제하고 검증** → 가드레일 (라우팅 + CI)
3. **실수에서 배워 규칙을 강화** → 진화
4. **쌓인 기술 부채를 자동 정리** → 위생

### 4-1. router.ts — 게이트웨이 (입력 분류기)

사용자의 입력을 LLM에 곧바로 보내는 대신, 라우터가 먼저 가로채서 **단순 대화인지, 엄격히 통제된 워크플로우를 트리거해야 하는지** 결정한다.

```
함수 route(사용자_입력):
    의도 = LLM으로_의도_분류(사용자_입력)

    만약 의도가 "일반_대화"이면:
        → 그냥 LLM에게 보내기

    만약 의도가 "코드_작성"이면:
        → 하네스 루프로 보내기

    만약 의도가 "모호함"이면:
        → 사용자에게 되묻기
```

> 모호한 요청은 되물어보고, 명확한 작업만 하네스 루프로 보낸다. 엉뚱한 방향으로 달려가는 걸 출발 전에 막는 것이다.
> 

### 4-2. context_manager.ts — 가림막 (점진적 공개)

아무런 제어 없는 LLM은 전체 코드베이스를 읽고 정작 중요한 제약 사항은 잊어버린다. **현재 AI에게 정확히 필요한 정보만 제공한다.**

```
함수 컨텍스트_구성(작업):
    관련_파일 = 작업에_필요한_파일만_선별()
    규칙 = CLAUDE.md에서_관련_규칙_추출()
    히스토리 = 최근_관련_대화만_요약()

    반환 {
        파일: 관련_파일,      // 전체가 아닌 필요한 것만
        규칙: 규칙,           // 항상 포함
        히스토리: 히스토리     // 압축된 형태로
    }
```

> 전체 코드베이스가 아니라, **지금 이 작업에 필요한 정보만 보여주는 것**이다.
> 

### 4-3. harness_loop.ts — 엔진 (자동 교정 장치) ⭐

하네스의 심장부. AI를 **계획 → 실행 → 검증 → 수정** 루프에 강제로 밀어넣는 상태 머신이다. 테스트를 통과하기 전까지 AI는 이 함수를 빠져나갈 수 없다.

```
함수 하네스_실행(작업, 최대_시도=5):
    시도 = 0

    반복 (시도 < 최대_시도):
        컨텍스트 = 컨텍스트_구성(작업)
        결과 = AI에게_코드_생성_요청(컨텍스트)

        검증 = 린터_실행(결과) + 테스트_실행(결과) + 타입체크(결과)

        만약 검증.통과:
            → 결과 반환 ✅

        그렇지 않으면:
            작업.에러_히스토리에_추가(검증.에러_메시지)
            시도 += 1
            → 에러를 포함해서 다시 루프 🔄

    → 최대 시도 초과 시 사람에게 에스컬레이션 🚨
```

> AI가 코드를 작성하면 자동으로 테스트를 돌리고, 실패하면 에러 메시지를 AI에게 다시 던져서 수정하게 한다. 사용자가 결과를 보기도 전에 AI가 스스로 오류를 수정하는 구조다.
> 

### 4-4. worker.ts — 실행부 (역할 분리)

코드를 작성하는 워커와 리뷰하는 워커를 **완전히 분리**하여 컨텍스트 오염을 방지한다.

```
함수 코드_생성_워커(컨텍스트):
    → LLM 호출 (시스템: "너는 코드 작성자다. 규칙을 따라라.")
    → 코드 반환

함수 코드_리뷰_워커(코드, 규칙):
    → 별도의 LLM 호출 (시스템: "너는 리뷰어다. 규칙 위반을 찾아라.")
    → 리뷰 결과 반환
```

> 같은 사람이 쓰고 검토하면 실수를 못 잡듯이, AI도 역할을 나눠야 품질이 올라간다.
> 

### 4-5. evolution.ts — 진화 (실수가 규칙이 된다)

하네스 루프에서 최대 시도를 초과하거나, 반복적으로 같은 종류의 에러가 발생하면 **자동으로 새 규칙을 생성**한다.

```
함수 진화_엔진(실패_로그):
    패턴 = 실패_로그에서_반복_패턴_분석()

    만약 패턴.유형 == "린트_규칙_위반_반복":
        → 새_린터_규칙 자동 추가
        → CLAUDE.md에 해당 금지 사항 추가

    만약 패턴.유형 == "아키텍처_위반_반복":
        → 구조 테스트 자동 생성

    만약 패턴.유형 == "권한_초과_시도":
        → 도구 경계 강화

    로그("새 규칙 추가됨: " + 패턴.설명)
```

> 에이전트가 실수할 때마다, 그 실수는 새로운 가드레일이 된다. 하네스가 스스로 진화하는 구조다.
> 

### 4-6. garbage_collector.ts — 위생 (쓰레기 수거)

에이전트가 코드를 빠르게 생성하면서 쌓이는 기술 부채를 **주기적으로 자동 정리**한다.

```
함수 가비지_컬렉션(코드베이스, 주기="매일"):
    // 죽은 코드 제거
    미사용_함수 = AST_분석으로_호출되지_않는_함수_탐지(코드베이스)
    미사용_import = 사용되지_않는_import_탐지(코드베이스)
    → 제거 PR 자동 생성

    // 중복 코드 정리
    중복_블록 = 유사도_분석으로_중복_코드_탐지(코드베이스)
    → 리팩토링 PR 자동 생성

    // 품질 지표 모니터링
    커버리지 = 테스트_커버리지_측정()
    만약 커버리지 < 임계값:
        → 알림 + 테스트 추가 요청
```

> 진화가 "앞으로 더 잘하게 만드는 것"이라면, 위생은 "이미 쌓인 쓰레기를 치우는 것"이다.
> 

### 전체 흐름

```
[실시간 흐름]
사용자 입력
  → [라우터] 의도 분류
    → [컨텍스트 매니저] 필요한 정보만 구성        ← 컨텍스트
      → [하네스 루프] 생성 → 검증 → 수정 반복     ← 가드레일
        → [코드 워커] 작성
        → [리뷰 워커] 검토
        → [CI 게이트] 린트 + 테스트
      → 통과 시 결과 반환 ✅
      → 실패 시 에러와 함께 재시도 🔄

[백그라운드 흐름]
  → [진화 엔진] 실패 로그 분석 → 새 규칙 자동 추가  ← 진화
  → [가비지 컬렉터] 주기적 코드 정리 → PR 생성      ← 위생
```

---

## 5. 구현 방식 비교 — 커스텀 TS 하네스 vs Claude Code

하네스 엔지니어링의 원리는 동일하지만, 구현 방식은 크게 두 가지로 나뉜다.

### 커스텀 TS 하네스 (직접 구축)

LLM API를 직접 호출하면서 라우터, 컨텍스트 매니저, 실행 루프, 워커를 **코드로 직접 작성**하는 방식이다.

**적합한 경우:**

- 사용자 대면 프로덕션 서비스를 만들 때
- 여러 LLM을 조합하거나 멀티 에이전트를 세밀하게 조율할 때
- 라우팅 로직을 비즈니스 요구사항에 맞게 커스터마이징할 때
- 작성 에이전트와 리뷰 에이전트의 완전한 분리가 필요할 때

**장점:** 완전한 제어권, 세밀한 커스터마이징, 프로덕션 배포 가능
**단점:** 구축 비용이 높고, 직접 유지보수해야 함

### Claude Code + CLAUDE.md (기존 도구 활용)

Claude Code에는 하네스의 핵심 요소가 **이미 내장**되어 있다.

| 하네스 역할 | Claude Code에서의 대응 |
| --- | --- |
| 컨텍스트 | `CLAUDE.md` 작성 → 에이전트가 자동 참조 |
| 가드레일 | 린터, 테스트, pre-commit hook + 내장 권한 시스템 |
| 진화 | 테스트 실패 → 에러 피드백 → 재시도 + 규칙 보강 |
| 위생 | dead code 제거, 중복 정리 등은 별도 설정 필요 |

**적합한 경우:**

- 개인 개발 또는 소규모 팀 프로젝트
- 빠르게 에이전트 기반 개발을 시작하고 싶을 때
- 이미 CI/CD 파이프라인이 있는 프로젝트에 에이전트를 도입할 때

**장점:** 즉시 시작 가능, 낮은 구축 비용, 빠른 이터레이션
**단점:** 세밀한 제어에 한계, 프로덕션 서비스 구축에는 부적합

### 핵심

특정 기술 스택이 중요한 게 아니다. **"실수가 구조적으로 반복 불가능하도록 시스템을 설계하라"**는 원칙이 중요하다. 그 원칙을 Claude Code 위에서 적용하든, 직접 TS로 짜든 둘 다 하네스 엔지니어링이다.
